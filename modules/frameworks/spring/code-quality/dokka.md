# Spring + Dokka

> Extends `modules/code-quality/dokka.md` with Spring Boot-specific integration.
> Generic Dokka conventions (installation, output formats, multi-module setup, source links) are NOT repeated here.

## Integration Setup

Apply Dokka in the root build; suppress internal Spring infrastructure packages from the public API docs:

```kotlin
// build.gradle.kts (root)
tasks.dokkaHtml {
    dokkaSourceSets {
        named("main") {
            perPackageOption {
                matchingRegex.set(".*\\.config\\..*")
                suppress.set(true)       // @Configuration classes are framework wiring, not API
            }
            perPackageOption {
                matchingRegex.set(".*\\.generated\\..*")
                suppress.set(true)       // annotation processor output
            }
        }
    }
}
```

## Framework-Specific Patterns

### REST Docs interop

Spring REST Docs generates AsciiDoc snippets from `MockMvc`/`WebTestClient` tests. Reference those snippets from KDoc to produce unified API docs:

```kotlin
/**
 * Creates a new order.
 *
 * REST Docs: see [create-order/curl-request.adoc] generated by `OrderControllerTest`.
 *
 * @param request validated request body — see `CreateOrderRequest`
 * @return 201 Created with `OrderResponse` body
 */
@PostMapping("/orders")
fun createOrder(@Valid @RequestBody request: CreateOrderRequest): ResponseEntity<OrderResponse>
```

Link AsciiDoc output into Dokka's custom assets directory to surface it alongside the API reference:

```kotlin
tasks.dokkaHtml {
    pluginsMapConfiguration.set(
        mapOf("org.jetbrains.dokka.base.DokkaBase" to
            """{ "customAssets": ["${projectDir}/build/generated-snippets"] }""")
    )
}
```

### Module doc structure

Use `Module.md` files to document the Spring module's responsibility and port/adapter boundaries:

```markdown
# Module order-service

Handles order lifecycle: creation, payment, fulfillment.

**Inbound ports:** `CreateOrderUseCase`, `CancelOrderUseCase`
**Outbound ports:** `OrderRepository`, `PaymentGateway`
**@Configuration classes** are internal framework wiring — see source for bean graph.
```

Reference in `build.gradle.kts`:

```kotlin
tasks.dokkaHtml {
    dokkaSourceSets {
        named("main") {
            includes.from("Module.md")
        }
    }
}
```

### `@Configuration` class docs

Suppress `@Configuration` classes from the public API site by convention (they expose no public domain API), but document them inline for maintainers:

```kotlin
/**
 * @suppress — internal Spring bean wiring.
 * Configures [OrderService] with [OrderRepository] and [PaymentGateway].
 */
@Configuration
internal class OrderConfig { ... }
```

Using `internal` visibility automatically suppresses from Dokka output without explicit `@suppress`.

## Additional Dos

- Mark `@Configuration` classes `internal` — they are invisible to Dokka and to other modules, reducing API surface noise.
- Include a `Module.md` per Spring module that maps inbound/outbound ports for hexagonal architecture clarity.
- Generate `dokkaHtmlMultiModule` in CI as part of the docs stage; publish to GitHub Pages or Confluence on main-branch merges.

## Additional Don'ts

- Don't document `@Bean` factory methods in KDoc unless the bean has non-obvious initialization side effects — Spring's context logs are sufficient for bean graph debugging.
- Don't let REST Docs AsciiDoc snippets drift from KDoc API descriptions — keep them in sync by referencing the same method from both.
- Don't suppress entire `web` or `api` packages from Dokka — controller and DTO interfaces are the primary external API contract.
