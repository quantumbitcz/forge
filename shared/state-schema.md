# State Schema

**Version:** 1.9.0
**Storage:** `.forge/state.json` (gitignored)
**Field reference:** see [`state-schema-fields.md`](state-schema-fields.md) for exhaustive field-by-field documentation, subsystem schemas (security injection, events.jsonl, checkpoints, stage notes, feedback, reports, orchestrator input, eval_run, prompt_compaction, speculation), and the full changelog.

## Directory Structure

```
.forge/
+-- state.json                          # Root pipeline state (one per run)
+-- checkpoint-{storyId}.json           # Per-story recovery checkpoint
+-- stage_N_notes_{storyId}.md          # Per-stage decisions and findings (N = 0-9)
+-- stage_final_notes_{storyId}.md      # Retrospective summary for the run
+-- feedback/
|   +-- summary.md                      # Consolidated feedback (created when >20 entries)
|   +-- {date}-{topic}.md              # Individual feedback files (YYYY-MM-DD-topic.md)
|   +-- archive/                        # Incorporated feedback moved here
+-- docs-index.json                     # Documentation index (fallback when Neo4j unavailable)
+-- evidence.json                       # Pre-ship verification evidence (created by fg-590)
+-- explore-cache.json                  # Codebase exploration cache (persisted across runs)
+-- plan-cache/                         # Cached PLAN outputs for similarity matching
|   +-- index.json                      # Plan cache index
|   +-- plan-{date}-{slug}.json         # Individual cached plans
+-- progress/                           # Background execution progress artifacts
|   +-- status.json                     # Current stage, score, ETA
|   +-- timeline.jsonl                  # Append-only event log
|   +-- stage-summary/                  # Completed stage summaries
|   +-- alerts.json                     # Issues requiring attention
+-- events.jsonl                        # Unified append-only event log (all pipeline events)
+-- decisions.jsonl                     # Branching decisions (backward-compat view of events.jsonl)
+-- automation-log.jsonl                # Event-driven automation log
+-- playbooks/                          # Reusable pipeline recipes (YAML definitions)
+-- playbook-analytics.json             # Playbook usage analytics (survives /forge-recover reset)
+-- agent-card.json                     # A2A agent card (generated by /forge-init)
+-- wiki/                               # Auto-generated codebase wiki
|   +-- .wiki-meta.json                 # Generation metadata
|   +-- index.md                        # Table of contents
|   +-- architecture.md                 # Detected architecture
|   +-- modules/                        # Per-domain module docs
+-- trust.json                          # Per-developer adaptive trust level (survives /forge-recover reset)
+-- run-history.db                       # Pipeline run history — SQLite FTS5 (survives /forge-recover reset)
+-- playbook-refinements/                # Playbook improvement proposals (survives /forge-recover reset)
+-- reports/
    +-- forge-{YYYY-MM-DD}.md       # Per-run retrospective report
    +-- recap-{YYYY-MM-DD}-{storyId}.md  # Human-readable run recap (by fg-710-post-run)
    +-- timeline-{storyId}.md            # Navigable pipeline timeline (by fg-710-post-run)
```

## Atomic writes

All writes to `.forge/state.json` are atomic. The `shared/forge-state-write.sh` helper performs a write-ahead-log (WAL) style update: it writes to a temporary sibling, `fsync`s, then `rename(2)`s over the target. Concurrent readers never observe a partial write.

Every write also increments the `_seq` monotonic counter (required field, starts at 1). Downstream consumers that cache state in memory compare `_seq` on reload to detect stale reads. A write that observes a newer `_seq` on disk than in its staged update aborts and retries — this is how we catch orchestrator ↔ agent interleaving when isolation assumptions leak.

Schema migrations (see [`state-schema-fields.md`](state-schema-fields.md#version-migration)) follow the same atomic-write path: the migrated payload is written with a bumped `version` and incremented `_seq`, never mutated in place.

## Concurrent-run lock

In single-feature mode, `.forge/.lock` is a per-repo lock file containing the holder PID. Acquired at PREFLIGHT, released at Stage 9 completion or on abort. A 24-hour stale timeout (checked against `.forge/state.json:lastCheckpoint`) lets subsequent runs reclaim the lock after a crashed orchestrator. Sprint mode uses per-run locks instead — see [`sprint-state-schema.md`](sprint-state-schema.md).

The lock is held by the orchestrator process only. Subagents dispatched by `fg-100-orchestrator` do not acquire or release it; they share the orchestrator's lock implicitly via the shared `.forge/` working set.
