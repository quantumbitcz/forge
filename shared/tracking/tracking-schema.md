# Tracking Schema

This document describes the kanban tracking data structures used by `forge` for
local ticket management. All files live under `.forge/tracking/` in the consuming
project root (gitignored by default).

---

## Directory Structure

```
.forge/tracking/
├── backlog/          # Tickets not yet started
├── in-progress/      # Tickets actively being worked
├── review/           # Tickets awaiting code review / PR feedback
├── done/             # Completed tickets (closed or merged)
├── counter.json      # Monotonic ID counter
└── board.md          # Auto-generated kanban summary table
```

Status directories are the canonical source of truth for a ticket's current
state. Moving a ticket file between directories is the primary state transition.

---

## Ticket File

Each ticket is stored as a single Markdown file named `{ID}-{slug}.md`
(e.g. `FG-007-add-user-auth.md`) inside the appropriate status directory.

### Filename Convention

```
{PREFIX}-{NNN}-{slug}.md
```

- `PREFIX` — project prefix from `counter.json` (default: `FG`)
- `NNN` — zero-padded to 3 digits for IDs 1-999; no padding for 1000+
- `slug` — kebab-case title, max 50 chars, special characters stripped

### Frontmatter

```yaml
---
id: FG-007
title: Add user authentication
type: feature          # feature | bug | chore | spike | refactor
status: backlog        # backlog | in-progress | review | done
priority: medium       # critical | high | medium | low
branch: ""             # git branch name (populated when work begins)
created: "2026-01-15T10:30:00Z"
updated: "2026-01-15T10:30:00Z"
linear_id: ""          # optional Linear issue ID (e.g. FG-42)
spec: ""               # path to spec file, if any
pr: ""                 # PR URL or number, populated during Ship stage
---
```

All fields are required in the frontmatter. Empty string `""` is the null value
for optional fields (`branch`, `linear_id`, `spec`, `pr`).

### Body

```markdown
## Description

Short description of the work to be done.

## Acceptance Criteria

- [ ] Criterion one
- [ ] Criterion two

## Stories

- [ ] Story / sub-task one
- [ ] Story / sub-task two

## Activity Log

- 2026-01-15T10:30:00Z — created (backlog)
- 2026-01-16T09:00:00Z — moved to in-progress
```

The **Activity Log** section is append-only. Every status transition appends a
timestamped line. `tracking-ops.sh` manages this automatically.

## Counter File

`counter.json` tracks the next ticket number and the project prefix.

```json
{
  "next": 1,
  "prefix": "FG"
}
```

- `next` — the next integer to assign (incremented after each allocation)
- `prefix` — uppercase string prepended to every ID (e.g. `FG`, `WP`, `BE`)

`init_counter` seeds this file. `next_id` reads, formats, and increments it
atomically within a single shell process.

---

## Board File

`board.md` is auto-generated by `generate_board` and should not be edited
manually. It provides a human-readable snapshot of the entire board.

```markdown
# Kanban Board

_Generated: 2026-01-16T09:00:00Z_

| ID | Title | Type | Priority | Status | Branch | PR |
|----|-------|------|----------|--------|--------|----|
| FG-001 | Bootstrap project | chore | high | done | feat/bootstrap | #1 |
| FG-007 | Add user authentication | feature | high | in-progress | feat/auth | |
| FG-012 | Fix login redirect bug | bug | critical | review | fix/login | #12 |
| FG-015 | Refactor session handling | refactor | medium | backlog | | |
```

Rows are sorted by status (backlog → in-progress → review → done) and then by
ID within each group.

---

## Status Transitions

Valid forward transitions:

```
backlog → in-progress → review → done
```

Valid back-transitions (for rejection or abort):

```
review      → in-progress   (PR feedback; resume work)
in-progress → backlog       (abort / deprioritise)
review      → backlog       (design rejected; start over)
done        → in-progress   (reopen; regression found)
```

Any transition not listed above is still allowed by `move_ticket` — the function
does not enforce direction. Governance is left to the calling skill/agent.

## Tracking Operations Reference

See `shared/tracking/tracking-ops.sh` for the implementation of:

| Function | Purpose |
|---|---|
| `slugify title [max_len]` | Convert title to kebab-case slug |
| `iso_now` | Current UTC timestamp in ISO 8601 |
| `init_counter tracking_dir [prefix]` | Seed `counter.json` |
| `next_id tracking_dir` | Allocate and return next ticket ID |
| `create_ticket tracking_dir title type priority [target_status]` | Create a new ticket file |
| `find_ticket tracking_dir ticket_id` | Locate ticket file across all status dirs |
| `move_ticket tracking_dir ticket_id new_status` | Transition ticket to a new status |
| `update_ticket_field tracking_dir ticket_id field value` | Update a single frontmatter field |
| `generate_board tracking_dir` | Regenerate `board.md` |
