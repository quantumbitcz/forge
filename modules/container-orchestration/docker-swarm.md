# Docker Swarm

## Overview

Docker Swarm is Docker's built-in container orchestration engine that turns a pool of Docker hosts into a single virtual host. It provides native clustering, service discovery, load balancing, rolling updates, and secrets management using the same Docker CLI and Compose file format that developers already know. Swarm mode is included in every Docker Engine installation — no additional software is required.

Use Docker Swarm when the team needs a simple container orchestrator that does not justify the operational complexity of Kubernetes. Swarm is well-suited for small to medium deployments (3-20 nodes), internal tools, staging environments, and teams where Docker expertise already exists but Kubernetes expertise does not. Swarm's key advantage is its simplicity: a `docker stack deploy` command with an existing `docker-compose.yml` file is all that is needed to deploy a multi-service application with load balancing, health checks, and rolling updates.

Do not use Docker Swarm for large-scale production deployments (50+ nodes) where Kubernetes' advanced scheduling, custom resource definitions, and ecosystem (Helm, operators, service mesh) provide significant operational advantages. Do not use Swarm when the team requires features that Swarm lacks: pod-level networking (sidecar containers), custom controllers, horizontal pod autoscaling, network policies, or a rich ecosystem of third-party operators. Do not use Swarm for stateful workloads that require sophisticated volume management — Kubernetes' StatefulSets and CSI drivers are significantly more mature.

**Swarm vs. Kubernetes comparison:**

| Feature | Docker Swarm | Kubernetes |
|---------|-------------|------------|
| Setup complexity | 1 command (`docker swarm init`) | Multi-component (API server, etcd, scheduler, controllers) |
| Learning curve | Low (reuses Docker CLI) | Steep (new concepts: pods, deployments, services, ingress) |
| Scaling | Manual (`docker service scale`) | Manual + HPA (Horizontal Pod Autoscaler) |
| Networking | Built-in overlay + routing mesh | CNI plugins (Calico, Cilium, Flannel) |
| Secrets | Built-in (`docker secret`) | Built-in (`kubectl create secret`) |
| Storage | Docker volumes (limited drivers) | CSI drivers (rich ecosystem) |
| Custom resources | Not supported | CRDs + operators |
| Ecosystem | Docker CLI only | Helm, operators, service mesh, policy engines |
| Max practical scale | ~100 nodes | ~5000 nodes |

## Architecture Patterns

### Cluster Initialization and Node Management

A Swarm cluster consists of manager nodes (which maintain cluster state and schedule services) and worker nodes (which run container workloads). The recommended topology is 3 or 5 manager nodes for fault tolerance (Raft consensus requires a quorum of `(n/2)+1` managers). Worker nodes join the cluster using a token generated by the init command.

**Cluster setup:**
```bash
# Initialize the first manager
docker swarm init --advertise-addr 10.0.1.10

# Output includes join tokens for workers and managers
# Save these tokens securely

# On worker nodes:
docker swarm join --token SWMTKN-1-abc123... 10.0.1.10:2377

# On additional manager nodes:
docker swarm join-token manager
# Copy the output command and run on the new manager node

# Verify cluster
docker node ls
```

**Node management:**
```bash
# Label nodes for placement constraints
docker node update --label-add zone=us-east-1a node-1
docker node update --label-add role=database node-2
docker node update --label-add gpu=true node-3

# Drain a node for maintenance (reschedules all tasks)
docker node update --availability drain node-2

# Promote a worker to manager
docker node promote node-3

# Demote a manager to worker
docker node demote node-2
```

### Stack Deployment

Stacks are the Swarm equivalent of Compose projects — they deploy a complete multi-service application from a Compose file. The `docker stack deploy` command reads a Compose file and creates services, networks, and volumes in the Swarm cluster. Stacks namespace all resources with the stack name as a prefix, preventing collisions between multiple deployments.

**Production stack file:**
```yaml
# stack.yml
services:
  api:
    image: registry.example.com/myapp:1.2.3
    deploy:
      replicas: 3
      update_config:
        parallelism: 1
        delay: 30s
        failure_action: rollback
        monitor: 60s
        max_failure_ratio: 0.3
        order: start-first
      rollback_config:
        parallelism: 1
        delay: 10s
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
        window: 120s
      placement:
        constraints:
          - node.role == worker
          - node.labels.zone == us-east-1a
        preferences:
          - spread: node.labels.zone
      resources:
        limits:
          cpus: "2.0"
          memory: 1G
        reservations:
          cpus: "0.5"
          memory: 512M
    ports:
      - target: 8080
        published: 8080
        protocol: tcp
        mode: ingress
    networks:
      - backend
    secrets:
      - db_password
      - api_key
    configs:
      - source: app_config
        target: /app/config/application.yml
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:8080/actuator/health"]
      interval: 15s
      timeout: 5s
      retries: 3
      start_period: 30s

  db:
    image: postgres:17-alpine
    deploy:
      replicas: 1
      placement:
        constraints:
          - node.labels.role == database
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - backend
    secrets:
      - db_password
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password

  nginx:
    image: nginx:1.27-alpine
    deploy:
      replicas: 2
      update_config:
        parallelism: 1
        delay: 10s
        order: start-first
    ports:
      - target: 80
        published: 80
        mode: ingress
      - target: 443
        published: 443
        mode: ingress
    configs:
      - source: nginx_conf
        target: /etc/nginx/nginx.conf
    networks:
      - backend

networks:
  backend:
    driver: overlay
    attachable: true

volumes:
  postgres_data:
    driver: local

secrets:
  db_password:
    external: true
  api_key:
    external: true

configs:
  app_config:
    file: ./config/application.yml
  nginx_conf:
    file: ./config/nginx.conf
```

```bash
# Deploy the stack
docker stack deploy -c stack.yml myapp

# List stacks
docker stack ls

# List services in a stack
docker stack services myapp

# View service logs
docker service logs myapp_api --follow --tail 100

# Scale a service
docker service scale myapp_api=5

# Remove a stack
docker stack rm myapp
```

### Rolling Updates

Swarm's rolling update mechanism replaces service tasks (containers) incrementally, allowing zero-downtime deployments with automatic rollback on failure. The `update_config` in the stack file controls every aspect of the update process.

```yaml
deploy:
  update_config:
    parallelism: 1          # Update 1 task at a time
    delay: 30s              # Wait 30s between task updates
    failure_action: rollback # Auto-rollback on failure
    monitor: 60s            # Monitor each task for 60s after start
    max_failure_ratio: 0.3  # Tolerate 30% task failures before rollback
    order: start-first      # Start new task before stopping old (blue-green per task)
```

**Update workflow:**
```bash
# Update service image
docker service update --image registry.example.com/myapp:1.3.0 myapp_api

# Update with environment variable change
docker service update --env-add NEW_VAR=value myapp_api

# Force update (re-deploy same config — useful for pulling updated images)
docker service update --force myapp_api

# Rollback to previous version
docker service rollback myapp_api

# Check update status
docker service inspect --pretty myapp_api
```

The `order: start-first` strategy is critical for zero-downtime deployments. It starts the new task and waits for it to become healthy before stopping the old task. This ensures that the service always has at least `replicas` instances running during the update. The alternative `stop-first` stops the old task before starting the new one, which temporarily reduces capacity but is necessary when host ports conflict.

### Configs and Secrets Management

Swarm provides first-class primitives for managing configuration data and sensitive credentials. Configs are mounted as files inside containers and can be updated by rotating to a new config version. Secrets are encrypted at rest, transmitted over TLS, and mounted as in-memory `tmpfs` files at `/run/secrets/`.

```bash
# Create secrets
echo "my-db-password" | docker secret create db_password -
docker secret create tls_cert ./certs/server.crt
docker secret create tls_key ./certs/server.key

# Create configs
docker config create app_config_v1 ./config/application.yml
docker config create nginx_conf_v1 ./config/nginx.conf

# Rotate a config (create new version, update service)
docker config create app_config_v2 ./config/application-v2.yml
docker service update \
  --config-rm app_config_v1 \
  --config-add source=app_config_v2,target=/app/config/application.yml \
  myapp_api

# List secrets and configs
docker secret ls
docker config ls
```

**Accessing secrets in applications:**
```yaml
# Stack file
services:
  app:
    secrets:
      - db_password
      - api_key
    environment:
      # Convention: *_FILE suffix tells the app to read from file
      DB_PASSWORD_FILE: /run/secrets/db_password
      API_KEY_FILE: /run/secrets/api_key
```

```kotlin
// Spring Boot reads *_FILE env vars natively via config trees
// application.yml:
// spring.datasource.password: ${DB_PASSWORD_FILE}
// Or read manually:
val password = File("/run/secrets/db_password").readText().trim()
```

## Configuration

### Development

For development, use `docker compose` instead of Swarm mode. Compose provides faster iteration (no cluster overhead), bind mounts for hot reload, and simpler debugging. Use Swarm only for testing deployment configurations and rolling updates.

```bash
# Initialize single-node Swarm for testing
docker swarm init

# Deploy stack locally
docker stack deploy -c stack.yml myapp-dev

# View logs
docker service logs -f myapp-dev_api

# Teardown
docker stack rm myapp-dev
docker swarm leave --force
```

### Production

Production Swarm clusters require careful node topology, network configuration, and monitoring.

```bash
# 3-manager, N-worker topology
# Manager 1 (initial):
docker swarm init --advertise-addr 10.0.1.10

# Manager 2, 3:
docker swarm join --token <manager-token> 10.0.1.10:2377

# Workers:
docker swarm join --token <worker-token> 10.0.1.10:2377

# Label nodes
docker node update --label-add zone=a manager-1
docker node update --label-add zone=b manager-2
docker node update --label-add zone=c manager-3

# Deploy with placement spread across zones
# (see stack.yml placement.preferences above)
docker stack deploy -c stack.yml myapp
```

**Manager node hardening:**
```bash
# Managers should not run application workloads
docker node update --availability drain manager-1
docker node update --availability drain manager-2
docker node update --availability drain manager-3
```

## Performance

**Routing mesh:** Swarm's built-in routing mesh forwards incoming requests on published ports to any node running the service, even if the request hits a node that is not running that service's tasks. This provides built-in load balancing but adds a network hop. For latency-sensitive workloads, use `mode: host` instead of `mode: ingress` to bypass the routing mesh and publish ports directly on the host — but this limits each service to one task per node on that port.

**Overlay network performance:** Swarm's overlay network uses VXLAN encapsulation, which adds ~50 bytes of overhead per packet and slightly increases latency compared to host networking. For throughput-critical internal services, consider using `mode: host` networking or placing communicating services on the same node with placement constraints.

**Resource management:** Always set `resources.limits` and `resources.reservations` in production. Reservations ensure the scheduler places tasks on nodes with sufficient resources. Limits prevent individual services from starving others. Without reservations, the scheduler may over-commit nodes, leading to OOM kills under load.

## Security

**Mutual TLS:** Swarm automatically encrypts all control plane traffic with mutual TLS. Data plane traffic (between services) uses the overlay network, which can optionally be encrypted with `--opt encrypted` on the network — but this adds CPU overhead for IPsec encryption. Evaluate whether data plane encryption is required based on the network trust model.

**Secrets:** Always use Docker secrets for sensitive data (passwords, API keys, TLS certificates). Secrets are encrypted at rest in the Raft log, transmitted only to nodes running tasks that need them, and mounted as in-memory tmpfs files that are never written to disk.

**Node certificates:** Swarm auto-rotates node certificates. Configure the rotation period:
```bash
docker swarm update --cert-expiry 720h  # 30 days
```

**Least privilege:** Drain manager nodes to prevent application workloads from running on them. Use placement constraints to control which nodes run sensitive services (databases, secret-handling services).

## Testing

**Stack file validation:**
```bash
# Validate stack file syntax
docker stack deploy --dry-run -c stack.yml myapp 2>&1

# Use docker compose config for YAML validation
docker compose -f stack.yml config --quiet
```

**Service health verification:**
```bash
# Check all tasks are running
docker stack services myapp --format "{{.Name}}: {{.Replicas}}"

# Check for failed tasks
docker service ps myapp_api --no-trunc --filter "desired-state=shutdown"

# Verify rolling update succeeded
docker service inspect --pretty myapp_api | grep -A5 "UpdateStatus"
```

**Integration testing with single-node Swarm:**
```bash
#!/usr/bin/env bash
set -euo pipefail

# Init single-node swarm
docker swarm init 2>/dev/null || true

# Deploy
docker stack deploy -c stack.yml test-myapp

# Wait for services
for i in $(seq 1 30); do
  READY=$(docker stack services test-myapp --format "{{.Replicas}}" | grep -c "1/1" || true)
  TOTAL=$(docker stack services test-myapp --format "{{.Replicas}}" | wc -l | tr -d ' ')
  [ "$READY" -eq "$TOTAL" ] && break
  sleep 2
done

# Run tests against the stack
curl -sf http://localhost:8080/actuator/health

# Cleanup
docker stack rm test-myapp
```

## Dos

- Use 3 or 5 manager nodes for production fault tolerance (Raft consensus requires a quorum).
- Drain manager nodes to prevent application workloads from running alongside cluster management.
- Use `update_config` with `failure_action: rollback` and `order: start-first` for zero-downtime rolling updates.
- Use Docker secrets for all sensitive data — they are encrypted at rest and in transit.
- Use Docker configs for non-sensitive configuration files that need version rotation.
- Set `resources.limits` and `resources.reservations` for every production service.
- Use placement constraints and preferences to spread services across availability zones.
- Label nodes by role, zone, and capability for flexible scheduling.

## Don'ts

- Do not run Swarm with a single manager in production — losing the only manager makes the cluster unrecoverable.
- Do not store secrets in environment variables or configs — use Docker secrets (encrypted, in-memory tmpfs).
- Do not use `mode: ingress` for latency-critical services without measuring the routing mesh overhead.
- Do not skip health checks — Swarm relies on them for rolling update decisions and restart policies.
- Do not use Swarm for workloads that require Kubernetes-specific features (CRDs, operators, HPA, network policies).
- Do not deploy stateful workloads (databases) on Swarm without careful volume management — Swarm's volume support is less mature than Kubernetes CSI.
- Do not allow application containers to run on manager nodes in production — drain all managers after cluster setup.
- Do not use `--force-new-cluster` unless recovering from a catastrophic quorum loss — it resets the Raft state.
