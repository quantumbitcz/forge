# k6 (Grafana k6) Best Practices

> Support tier: contract-verified

## Overview
k6 is an open-source load testing tool using JavaScript/TypeScript scripts. Use it for performance testing, stress testing, and soak testing of APIs, microservices, and websites. k6 excels at developer-friendly scripting, CI/CD integration, and Grafana Cloud k6 for distributed load generation. Avoid it for browser-based E2E testing (use Playwright/Cypress), or when JMeter's GUI-based test design is preferred by QA teams.

## Conventions

### Test Structure
```javascript
import http from "k6/http";
import { check, sleep } from "k6";
import { Rate } from "k6/metrics";

const errorRate = new Rate("errors");

export const options = {
  stages: [
    { duration: "1m", target: 50 },   // ramp up
    { duration: "3m", target: 50 },   // steady state
    { duration: "1m", target: 0 },    // ramp down
  ],
  thresholds: {
    http_req_duration: ["p(95)<500", "p(99)<1000"],
    errors: ["rate<0.01"],
  },
};

export default function () {
  const res = http.get("https://api.example.com/products");
  check(res, {
    "status is 200": (r) => r.status === 200,
    "response time < 500ms": (r) => r.timings.duration < 500,
  }) || errorRate.add(1);
  sleep(1);
}
```

### Lifecycle Hooks
```javascript
export function setup() {
  const loginRes = http.post("https://api.example.com/auth/login", JSON.stringify({ email: "test@example.com", password: "test" }), { headers: { "Content-Type": "application/json" } });
  return { token: loginRes.json("token") };
}

export default function (data) {
  const params = { headers: { Authorization: `Bearer ${data.token}` } };
  http.get("https://api.example.com/orders", params);
}

export function teardown(data) {
  // cleanup
}
```

## Configuration

```bash
# Run locally
k6 run --vus 50 --duration 5m script.js

# Run with environment variables
k6 run -e BASE_URL=https://staging.example.com script.js

# Output to Grafana Cloud k6
k6 run --out cloud script.js
```

## Dos
- Define `thresholds` in every test — they make tests pass/fail programmatically in CI.
- Use `check()` for response validation — it reports pass/fail rates without stopping the test.
- Use `sleep()` between requests to simulate real user think time.
- Use `stages` for realistic ramp-up/ramp-down patterns — sudden load spikes aren't representative.
- Use `setup()` for one-time initialization (auth tokens, test data) — it runs once before VUs start.
- Use custom metrics (`Rate`, `Counter`, `Gauge`, `Trend`) for business-specific KPIs.
- Run load tests in CI/CD against staging — catch performance regressions before production.

## Don'ts
- Don't skip `thresholds` — without them, load tests don't produce a pass/fail verdict.
- Don't omit `sleep()` — zero think time generates unrealistic burst traffic.
- Don't run load tests against production without coordination — it can trigger rate limits or outages.
- Don't use k6 for functional E2E testing — it's a load testing tool, not a browser automation tool.
- Don't hardcode URLs — use environment variables (`__ENV.BASE_URL`) for portability.
- Don't test from a single machine for high-concurrency scenarios — use distributed execution (k6 Cloud).
- Don't ignore the `http_req_duration` p99 — p50 hides tail latency issues that affect real users.
