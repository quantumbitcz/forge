```markdown
## Confidence-Weighted Scoring

When findings include a confidence field (`confidence:HIGH`, `confidence:MEDIUM`, `confidence:LOW`), the scoring formula applies a confidence multiplier to the severity deduction:

```
deduction = severity_weight * confidence_multiplier
```

| Confidence | Multiplier | Effect |
|------------|-----------|--------|
| `HIGH` | 1.0 | Full deduction (default when confidence omitted) |
| `MEDIUM` | 1.0 | Full deduction (reviewer stands behind it) |
| `LOW` | 0.5 | Half deduction (reviewer uncertain) |

### Example

| Finding | Severity | Confidence | Raw Deduction | Weighted Deduction |
|---------|----------|-----------|--------------|-------------------|
| SEC-AUTH-001 | CRITICAL | HIGH | -20 | -20 |
| ARCH-LAYER-002 | WARNING | MEDIUM | -5 | -5 |
| QUAL-NAME-003 | INFO | LOW | -2 | -1 |

Score: `100 - 20 - 5 - 1 = 74` (vs. `100 - 20 - 5 - 2 = 73` without confidence weighting)

### Backward Compatibility

When confidence is omitted, the default is `HIGH` (multiplier 1.0). This preserves the existing scoring behavior for any findings that don't include a confidence field.

### Routing by Confidence

The quality gate uses confidence for dispatch routing:
- **HIGH/MEDIUM findings:** Auto-dispatched to implementer for fixing
- **LOW findings:** Flagged for human review, NOT auto-dispatched to implementer. Included in stage notes and recap but excluded from fix cycles.

The `decision_quality.findings_with_low_confidence` counter in `state.json` tracks how many LOW-confidence findings were produced per run. This feeds into the retrospective's reviewer accuracy analysis.
```

```markdown
All review agents MUST include the `confidence` field in every finding. The field is the 6th pipe-delimited value in the standard finding format (see `shared/checks/output-format.md`).

| Confidence | When to Use |

- Default to `confidence:HIGH` when certain. Do not over-use LOW to hedge — LOW findings receive half scoring weight and are excluded from fix cycles.

- The confidence field applies to all categories including SCOUT-* (though SCOUT findings are already zero-scored).
```

```markdown
## Deliberation Response Format

When the quality gate detects conflicting findings at the same `(file, line)` and `quality_gate.deliberation` is enabled, it re-dispatches both originating reviewers with a narrow deliberation prompt. Each reviewer responds with one of:

| Response | Meaning | Effect |
|----------|---------|--------|
| `MAINTAIN` | Keep finding as-is, add reasoning | Finding survives with original severity |
| `REVISE` | Adjust severity, explain why | Finding updated with new severity |
| `WITHDRAW` | Concede to other agent's perspective | Finding removed |

### Deliberation Prompt Format (received by reviewers)

```
Your finding conflicts with another reviewer's finding at the same location.

YOUR FINDING:
  file:line | CATEGORY | SEVERITY | CONFIDENCE | description | fix_hint

CONFLICTING FINDING (from {other_agent}):
  file:line | CATEGORY | SEVERITY | CONFIDENCE | description | fix_hint

Review both findings and respond with exactly one of:
- MAINTAIN: {reasoning why your finding should stand}
- REVISE severity to {WARNING|INFO}: {reasoning for downgrade}
- WITHDRAW: {reasoning why the other finding is more appropriate}
```

### Constraints

- Max 1 deliberation round per conflict (no back-and-forth)
- Only triggered for conflicts involving at least one WARNING or CRITICAL finding
- 60-second timeout per reviewer response. On timeout, original finding stands unchanged.
- Deliberation is disabled by default (`quality_gate.deliberation: false`)
```

```markdown
# Mutation Analyzer

Generates targeted code mutants for changed files and runs the existing test suite against each mutant. Surviving mutants (tests didn't catch them) become `TEST-MUTATION-*` findings.

Dispatched by `fg-500-test-gate` as a sub-agent within Stage 5 (VERIFY) after Phase B passes. Findings are included in `stage_5_notes` and routed to the quality gate by the orchestrator.

## Input

Received from `fg-500-test-gate` dispatch prompt:
- Changed files list (from implementation stage)
- Test commands (build, test run commands from `forge.local.md`)
- Mutation categories to apply (from `mutation_testing.categories` in config)
- Max mutants per file (from `mutation_testing.max_mutants_per_file`, default 5)

## Mutation Categories

| Category | Description | Example Mutations |
|----------|------------|-------------------|
| `boundary_conditions` | Off-by-one errors, boundary comparisons | `<=` → `<`, `>=` → `>`, `i < n` → `i < n-1` |
| `null_handling` | Removed null/nil/undefined checks | Remove `if (x == null) return`, remove `?.` operator |
| `error_paths` | Exception handling changes | Remove catch block body, change exception type, remove throw |
| `logic_inversions` | Boolean logic changes | `&&` → `\|\|`, `!` removal, `true` → `false` |

## Process

1. **Select mutation targets:** Read each changed file, identify lines/expressions that match the configured categories
2. **Generate mutants:** For each target, create 1 mutant (minimal change). Max `max_mutants_per_file` per file.
3. **Run tests per mutant:**
   a. Apply the mutation (Edit tool)
   b. Run the test suite (Bash tool with test command)
   c. Record result: KILLED (test failed = good), SURVIVED (test passed = gap), TIMEOUT, ERROR
   d. Revert the mutation (Edit tool to restore original)
4. **Classify results** into finding categories

## Finding Categories

| Code | Severity | When | Description |
|------|----------|------|-------------|
| `TEST-MUTATION-SURVIVE` | WARNING | Tests pass with mutant | Test gap — mutation was not detected by any test |
| `TEST-MUTATION-TIMEOUT` | INFO | Tests timeout with mutant | Likely infinite loop — mutation probably meaningful (good sign) |
| `TEST-MUTATION-EQUIVALENT` | INFO | Mutant produces identical behavior | Mutation doesn't change semantics — not a test gap |

## Output Format

Findings in standard format per `shared/checks/output-format.md`:

```
file:line | TEST-MUTATION-SURVIVE | WARNING | confidence:HIGH | Boundary mutant survived: changed `<=` to `<` at line 42, no test detected the difference | Add test for boundary value at exactly n
file:line | TEST-MUTATION-EQUIVALENT | INFO | confidence:HIGH | Equivalent mutant: removing null check at line 15 has no effect because caller already validates | No action needed — document as verified
```

## Constraints

- **Read-only except for temporary mutations:** Mutations are applied and immediately reverted. No permanent changes to source files.
- **Scope:** Only mutate files in the changeset (not the entire codebase)
- **Budget:** Max `max_mutants_per_file` (default 5) mutations per file. Max 30 total mutations per run.
- **Timeout per mutant:** If test suite doesn't complete within 2x the normal test duration, classify as TIMEOUT
- **Worktree safety:** All mutations happen in the worktree. User's tree is never touched.

## Configuration

From `forge-config.md` (passed via dispatch prompt):

    mutation_testing:
      enabled: true
      scope: changed_files_only
      max_mutants_per_file: 5
      severity_on_surviving: WARNING
      categories:
        - boundary_conditions
        - null_handling
        - error_paths
        - logic_inversions

## Forbidden Actions

Per `shared/agent-defaults.md` Standard Reviewer Constraints, plus:
- Do NOT leave mutations in place after analysis (always revert)
- Do NOT generate mutants for test files (only source files)
- Do NOT modify build configuration or test configuration
- Do NOT create new test files (only analyze existing test coverage)
```

```markdown
# Visual Verification

Defines how frontend agents use Playwright MCP for screenshot-based UI verification. Conditional on Playwright MCP availability and frontend files in the changeset.

## Prerequisites

Visual verification activates when ALL of these conditions are met:
1. Playwright MCP is available (`state.json.integrations.playwright.available == true`)
2. Frontend files are in the changeset (detected by file extensions: `.tsx`, `.jsx`, `.vue`, `.svelte`, `.html`, `.css`, `.scss`)
3. `visual_verification.enabled` is `true` in config
4. `visual_verification.dev_server_url` is configured

If any condition is not met, visual verification is skipped with no finding (silent skip — not even INFO).

## Screenshot Strategy

### Breakpoints

Three standard breakpoints for responsive verification:

| Name | Width | Represents |
|------|-------|-----------|
| mobile | 375px | iPhone SE / small phone |
| tablet | 768px | iPad / tablet |
| desktop | 1440px | Standard desktop |

Configurable via `visual_verification.breakpoints` (array of integers).

### Page Detection

Pages to screenshot are determined by:
1. `visual_verification.pages` config (explicit list) — highest priority
2. Auto-detection from changed routes (if framework supports file-based routing: Next.js, SvelteKit, Nuxt)
3. Fallback: screenshot the root URL (`/`) only

### Capture Process

For each page at each breakpoint:
1. Navigate to `{dev_server_url}{page_path}`
2. Wait for network idle (Playwright's `waitUntil: 'networkidle'`)
3. Take full-page screenshot
4. Store screenshot reference for comparison

## Verification Modes

### fg-413 (Frontend Reviewer) — Code + Visual Review

During REVIEW stage:
1. Take screenshots at all breakpoints for changed pages
2. Analyze screenshots for:
   - Layout breaks (elements overlapping, overflowing, missing)
   - Responsive behavior (content reflows correctly across breakpoints)
   - Contrast issues (text readability against backgrounds)
   - Visual consistency (spacing, alignment, typography)
3. Report findings as `FE-VISUAL-*` categories

### fg-320 (Frontend Polisher) — Before/After Comparison

During IMPLEMENT stage:
1. Take "before" screenshots at all breakpoints BEFORE applying polish
2. Apply polish changes
3. Take "after" screenshots at same breakpoints
4. Compare before/after for:
   - Design token compliance (colors, spacing match design system)
   - Visual hierarchy improvements
   - Animation/transition smoothness
5. Include before/after evidence in stage notes

## Finding Categories

| Code | Severity | Description |
|------|----------|-------------|
| `FE-VISUAL-REGRESSION` | WARNING | Unexpected visual change detected (layout shift, element missing) |
| `FE-VISUAL-RESPONSIVE` | WARNING | Layout breaks at a specific breakpoint |
| `FE-VISUAL-CONTRAST` | WARNING | Text contrast ratio below WCAG AA threshold |
| `FE-VISUAL-FIDELITY` | INFO | Visual output deviates from design spec (when Figma context available) |

## Graceful Degradation

| Failure | Behavior |
|---------|----------|
| Playwright MCP unavailable | Silent skip — no findings, no stage notes |
| Dev server not running | INFO finding: "Dev server at {url} unreachable — visual verification skipped" |
| Screenshot capture fails | INFO finding per failed page: "Screenshot failed for {page} at {breakpoint}" |
| All screenshots fail | WARNING finding: "Visual verification failed completely — investigate dev server" |

## Configuration

In `forge-config.md`:

    visual_verification:
      enabled: true
      dev_server_url: "http://localhost:3000"
      breakpoints: [375, 768, 1440]
      pages: []

| Parameter | Range | Default | Description |
|-----------|-------|---------|-------------|
| `enabled` | boolean | `true` | Enable/disable visual verification |
| `dev_server_url` | URL string | `""` (empty = skip) | Dev server URL |
| `breakpoints` | int[] | `[375, 768, 1440]` | Viewport widths for screenshots |
| `pages` | string[] | `[]` (auto-detect) | Explicit page paths to screenshot |
```

```markdown
# LSP Integration

Defines how agents use the Language Server Protocol (LSP) tool for compiler-level code understanding. LSP provides precise symbol resolution, type checking, and diagnostics that surpass grep/glob-based analysis.

## Availability

The `LSP` tool is available in Claude Code when a language server is running for the project's language. Availability is detected at runtime — agents should attempt LSP operations and fall back gracefully.

## When to Use LSP

| Operation | Without LSP | With LSP |
|-----------|------------|----------|
| Find all references to a symbol | `Grep` (pattern matching, may miss/over-match) | `LSP` find-references (compiler-precise) |
| Go to definition | `Grep` + file reading (heuristic) | `LSP` go-to-definition (exact) |
| Check for unused code | Pattern matching (unreliable for re-exports) | `LSP` diagnostics (compiler-verified) |
| Verify import boundaries | `Grep` for import statements | `LSP` workspace symbols + dependency graph |
| Type checking | Manual reading (error-prone) | `LSP` diagnostics (compiler errors/warnings) |

### Agents That Benefit

| Agent | LSP Usage |
|-------|-----------|
| `fg-412-architecture-reviewer` | Precise dependency graph via find-references. Verify layer boundaries without false positives from string matching. |
| `fg-416-backend-performance-reviewer` | Type-aware analysis — distinguish lazy vs eager collections, identify generic type constraints that affect performance. |
| `fg-300-implementer` | Accurate refactoring via go-to-definition and find-references. Rename symbols safely. |
| `fg-410-code-reviewer` | Compiler diagnostics as additional finding source. Unused variables, unreachable code. |
| EXPLORE stage | Symbol-level codebase map — exports, dependencies, module boundaries with precision. |

## Graceful Degradation

LSP is ALWAYS optional. If the LSP tool is unavailable or returns an error:
1. Fall back to Grep/Glob/Read-based analysis (pre-v1.18 behavior)
2. Do NOT emit an INFO finding about LSP unavailability (silent fallback)
3. Do NOT reduce analysis scope or skip checks — perform the same checks using text-based tools
4. Log in stage notes: "LSP: unavailable, using text-based analysis" (for telemetry only)

## Supported Languages

LSP integration is most effective for languages with mature language servers:

| Language | Server | LSP Quality |
|----------|--------|-------------|
| TypeScript | tsserver | Excellent — full type system |
| Kotlin | kotlin-language-server | Good — requires Gradle project |
| Python | pyright / pylsp | Good — type stubs improve accuracy |
| Rust | rust-analyzer | Excellent — full type system |
| Go | gopls | Excellent — fast, comprehensive |
| Java | Eclipse JDT | Good — requires build system |
| C# | OmniSharp | Good — requires .NET SDK |
| Swift | sourcekit-lsp | Moderate — Xcode project required |

## Configuration

In `forge-config.md`:

    lsp:
      enabled: true
      languages:
        - typescript
        - kotlin
        - python
        - rust
        - go

| Parameter | Range | Default | Description |
|-----------|-------|---------|-------------|
| `lsp.enabled` | boolean | `true` | Enable/disable LSP usage across agents |
| `lsp.languages` | string[] | auto-detected | Languages to attempt LSP for |

When `lsp.enabled` is `false`, agents never attempt LSP operations (pure text-based analysis).

## Agent Integration Pattern

Agents that support LSP should follow this pattern:

1. Check if LSP is likely available (project has relevant language files)
2. Attempt the LSP operation (e.g., find-references)
3. If successful: use the precise result
4. If failed: fall back to Grep/Glob equivalent
5. Do NOT block on LSP — set a 10-second timeout per operation

Example in agent workflow:

    # Preferred: LSP find-references for precise dependency check
    # Fallback: Grep for import statements
    Try LSP find-references for {symbol}
    If LSP unavailable or times out:
      Grep for "{symbol}" in project files
      Filter by import/require patterns
```
