# DVC — Data Version Control

## Overview

DVC tracks large files, datasets, and ML models alongside Git without storing them in the repository. It defines reproducible ML pipelines via `dvc.yaml` and manages remote storage (S3, GCS, Azure, SSH) for data artifacts. DVC enables experiment comparison, data lineage tracking, and cache-efficient pipeline re-execution by detecting which stages have changed inputs.

## Architecture Patterns

### Data Tracking

```bash
dvc init
dvc add data/training.csv
git add data/training.csv.dvc data/.gitignore
git commit -m "track training dataset v1"
dvc push
```

Every large file (datasets, models, embeddings) must be tracked via `dvc add` — never commit binary files directly to Git. The `.dvc` file stores the MD5 hash and remote location.

### Pipeline Definitions

```yaml
# dvc.yaml
stages:
  preprocess:
    cmd: python src/preprocess.py
    deps:
      - src/preprocess.py
      - data/raw/
    outs:
      - data/processed/
  train:
    cmd: python src/train.py
    deps:
      - src/train.py
      - data/processed/
    params:
      - train.learning_rate
      - train.epochs
    outs:
      - models/model.pkl
    metrics:
      - metrics.json:
          cache: false
    plots:
      - plots/loss.csv:
          x: epoch
          y: loss
```

Define every pipeline stage with explicit `deps`, `outs`, `params`, and `metrics`. DVC skips unchanged stages on re-execution. Missing `deps` declarations cause silent cache hits on stale data.

### Remote Storage

```bash
dvc remote add -d myremote s3://my-bucket/dvc-store
dvc remote modify myremote region us-east-1
dvc push
```

Configure a default remote for team collaboration. All team members must have access to the same remote for `dvc pull` to work.

## Configuration

- Initialize DVC in the Git repository root with `dvc init` — the `.dvc/` directory stores configuration.
- Set a default remote (`dvc remote add -d`) pointing to shared object storage.
- Use `params.yaml` for hyperparameters referenced by `dvc.yaml` — enables `dvc params diff` for experiment comparison.
- Configure `.dvcignore` to exclude temporary files and build artifacts from DVC tracking.
- Use `dvc.lock` (auto-generated) to pin exact hashes of all stage inputs and outputs — commit this file to Git.

## Performance

- DVC uses content-addressable storage with MD5 hashing — duplicate files are stored once regardless of location.
- Use `dvc checkout` after `git checkout` to sync data files with the correct branch version.
- Pipeline stages with unchanged inputs are skipped automatically — declare all dependencies accurately to benefit from caching.
- For large datasets, use `dvc import` with `--rev` to pin a specific version from an external repository.
- Prefer `dvc push --jobs N` for parallel uploads to remote storage.

## Security

- Never commit remote storage credentials to Git — use environment variables or IAM roles.
- Ensure DVC remote storage has appropriate access controls — DVC does not manage permissions.
- Audit `.dvc` files in PRs to detect unexpected data changes (hash changes indicate data modifications).
- Use signed URLs or IAM policies for temporary access to DVC remotes in CI/CD pipelines.

## Testing

- Validate pipeline reproducibility: `dvc repro` on a clean checkout should produce identical outputs (matching `dvc.lock` hashes).
- Test that `dvc pull` retrieves all required data files before running training scripts.
- Verify `dvc status` reports no changes after a successful `dvc repro` — any diff indicates missing dependencies.
- Use `dvc params diff` to compare hyperparameters between experiment branches.

## Dos
- Track all large files (datasets, models, embeddings) with `dvc add` — keep Git clean of binaries.
- Declare all dependencies (`deps`) in `dvc.yaml` stages — missing deps cause incorrect cache hits.
- Commit `dvc.lock` to Git after every `dvc repro` — it captures the exact state of all artifacts.
- Use `params.yaml` for hyperparameters and reference them in `dvc.yaml` with `params:`.
- Configure a shared remote storage for team collaboration.
- Pin dataset versions with `dvc import --rev` for reproducible external data references.

## Don'ts
- Don't commit large binary files directly to Git — use `dvc add` and track the `.dvc` file instead.
- Don't manually edit `dvc.lock` — it is auto-generated by `dvc repro`.
- Don't omit `deps` from pipeline stages — DVC cannot detect changes to undeclared dependencies.
- Don't store credentials in `.dvc/config` — use `.dvc/config.local` (gitignored) or environment variables.
- Don't skip `dvc push` after adding or updating tracked files — team members will fail on `dvc pull`.
- Don't mix DVC-tracked and Git-tracked versions of the same file — choose one tracking mechanism.
