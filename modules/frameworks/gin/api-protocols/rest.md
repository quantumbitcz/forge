# Gin REST — API Protocol Binding

## Integration Setup
- `github.com/gin-gonic/gin`; use `gin.New()` + explicit middleware over `gin.Default()` for production
- Validation: built-in `binding` tags via `go-playground/validator`; custom validators registered via `binding.Validator`
- OpenAPI/Swagger: `github.com/swaggo/swag` + `github.com/swaggo/gin-swagger`; generate with `swag init`

## Framework-Specific Patterns
- Register routes on a `*gin.RouterGroup`: `v1 := r.Group("/api/v1")` then `v1.GET("/users/:id", handler)`
- Handler signature: `func(c *gin.Context)`; use `c.ShouldBindJSON(&req)` for body binding with validation
- Respond with `c.JSON(http.StatusOK, gin.H{"data": result})` or a typed response struct
- Error handling: define middleware that reads errors from `c.Errors.Last()` and writes a uniform JSON response
- Middleware: `r.Use(middleware)` at router level; inject dependencies via closure or `gin.Context` keys
- Dependency injection: pass service interfaces into handler factory functions via closure

## Scaffolder Patterns
```
internal/
  handler/
    user_handler.go          # handler factory func returning gin.HandlerFunc
    dto/
      create_user_request.go # binding-tagged structs
      user_response.go
  middleware/
    auth.go
    error_handler.go         # reads c.Errors, writes JSON response
  router/
    router.go                # gin.Engine setup, route registration
docs/
  swagger.yaml               # generated by swag init
```

## Dos
- Use `ShouldBind` variants (not `MustBind`) to handle validation errors gracefully
- Set `gin.SetMode(gin.ReleaseMode)` in production; `DebugMode` logs route info on startup
- Use `c.AbortWithStatusJSON` in middleware to halt the chain cleanly
- Add `swag` annotations on handler functions and run `swag init` in CI to keep docs current

## Don'ts
- Don't use `gin.Default()` in production — `gin.New()` with explicit `Recovery()` and `Logger()` is preferred
- Don't store mutable shared state in `gin.Context` keys across goroutines
- Don't call `c.JSON` multiple times in a single handler path — it sets the response twice
- Don't put database access inside handler functions — route through a service layer
