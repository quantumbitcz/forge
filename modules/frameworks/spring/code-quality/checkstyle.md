# Spring + Checkstyle

> Extends `modules/code-quality/checkstyle.md` with Spring Boot-specific integration.
> Generic Checkstyle conventions (installation, rule categories, CI integration) are NOT repeated here.

## Integration Setup

Checkstyle is relevant for Spring Java code. Kotlin modules should use detekt + ktlint instead. Apply selectively to Java source sets:

```kotlin
// build.gradle.kts
plugins {
    checkstyle
}

checkstyle {
    toolVersion = "10.17.0"
    configFile = file("$rootDir/config/checkstyle/checkstyle.xml")
    isIgnoreFailures = false
    maxWarnings = 0
}

// Apply only to Java sources — not Kotlin
tasks.checkstyleMain { source = fileTree("src/main/java") }
tasks.checkstyleTest { source = fileTree("src/test/java") }
```

## Framework-Specific Patterns

### Line length for annotation-heavy code

Spring Java code uses stacked annotations (`@RestController`, `@RequestMapping`, `@PreAuthorize`, `@Validated`) that push method signatures past 80 or 100 characters. Set a permissive but bounded limit:

```xml
<!-- config/checkstyle/checkstyle.xml -->
<module name="LineLength">
    <property name="max" value="140"/>
    <property name="ignorePattern" value="^package.*|^import.*|a href|href|http://|https://|ftp://"/>
</module>
```

Allow long annotation expressions without flagging:

```xml
<module name="LineLength">
    <property name="ignorePattern"
              value="@\w+(\(.*\))?|^import|^package"/>
</module>
```

### Exclusions for generated configuration

Spring application configs and migration scripts in the `src/main` tree are not Java source — exclude them:

```xml
<!-- checkstyle-suppressions.xml -->
<suppressions>
  <!-- Generated by Spring Initializr or OpenAPI codegen -->
  <suppress files="[\\/]generated[\\/]" checks=".*"/>
  <!-- Spring Boot main application class -->
  <suppress files="Application\.java$" checks="HideUtilityClassConstructor"/>
  <!-- Spring configuration classes — allow multiple annotations -->
  <suppress files=".*Config\.java$" checks="VisibilityModifier"/>
</suppressions>
```

Reference suppressions file in main config:

```xml
<module name="SuppressionFilter">
    <property name="file" value="${config_loc}/checkstyle-suppressions.xml"/>
</module>
```

### Javadoc requirements for public APIs

Require Javadoc on public Spring controllers and service interfaces, but not on `@Configuration` beans:

```xml
<module name="JavadocMethod">
    <property name="scope" value="public"/>
    <property name="allowMissingParamTags" value="false"/>
    <property name="tokens" value="METHOD_DEF"/>
</module>

<!-- Suppress Javadoc check for @Configuration classes -->
<module name="SuppressWithNearbyCommentFilter">
    <property name="commentFormat" value="@Configuration"/>
    <property name="checkFormat" value="JavadocMethod"/>
</module>
```

## Additional Dos

- Set `max line length` to 140 for Spring projects — annotation stacking makes 100/120 impractical.
- Suppress `HideUtilityClassConstructor` for `*Application.java` — Spring Boot main classes require a public constructor.
- Restrict Checkstyle to Java source sets only; use detekt + ktlint for Kotlin source.

## Additional Don'ts

- Don't apply Checkstyle to `build/generated/` — OpenAPI and Hibernate metamodel generators emit non-compliant Java intentionally.
- Don't require `@author` tags in Javadoc — Spring teams use Git history for attribution, not inline tags.
- Don't enforce `VisibilityModifier` on `@ConfigurationProperties` classes — Spring needs setters or non-final fields for binding.
