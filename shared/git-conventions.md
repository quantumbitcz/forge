# Git Conventions

Shared reference document for branch naming, commit formatting, and hook detection used by `fg-100-orchestrator`, `fg-600-pr-builder`, and `/forge`.

Configuration lives in the `git:` section of `forge.local.md`. All fields have hardcoded defaults; omit to accept defaults.

---

## Branch Naming

### Default Template

```
{type}/{ticket}-{slug}
```

- `{type}` — branch type prefix (see type mapping below)
- `{ticket}` — ticket identifier sourced from `ticket_source` (omitted when no ticket resolves)
- `{slug}` — lowercase kebab-case summary of the requirement, truncated to `slug_max_length`

Examples:
```
feat/FG-42-add-payment-flow
fix/PROJ-7-null-pointer-on-login
refactor/extract-domain-events
chore/bootstrap-new-service
```

### Configuration (`forge.local.md` `git:` section)

```yaml
git:
  branch_template: "{type}/{ticket}-{slug}"   # default
  branch_types:                                # override to restrict allowed types
    - feat
    - fix
    - refactor
    - chore
    - migrate
  slug_max_length: 40                          # characters, default 40
  ticket_source: auto                          # auto | linear | kanban | none
```

### `ticket_source: auto` Resolution

When `ticket_source` is `auto`, the orchestrator resolves the ticket prefix using this priority order:

1. **Linear ID** — if Linear MCP is available and a story ID exists in `state.json.linear.story_ids`, use the first entry (e.g., `FG-42`)
2. **Kanban ticket ID** — if a kanban ticket is linked in `state.json.kanban.ticket_id`, use that value
3. **No prefix** — if neither is available, the `{ticket}-` segment is omitted entirely, yielding `{type}/{slug}`

### Type Mapping

Pipeline mode → branch type:

| Pipeline mode / trigger | Branch type |
|-------------------------|-------------|
| Standard (new feature)  | `feat`      |
| Bugfix / hotfix         | `fix`       |
| Migration mode          | `migrate`   |
| Bootstrap mode          | `chore`     |
| Explicit refactor       | `refactor`  |

The orchestrator selects the type automatically from `state.json.mode` and intent signals. Consumers may override via `branch_types` to restrict the allowed set.

## Commit Format

### Default Format

Conventional Commits:

```
{type}({scope}): {description}
```

- `{type}` — one of the configured `commit_types`
- `{scope}` — optional architectural scope (omit when `require_scope: false`)
- `{description}` — imperative mood, lowercase, no trailing period, max `max_subject_length` characters

Examples:
```
feat(payments): add stripe checkout session creation
fix(auth): handle expired jwt on refresh endpoint
chore: bump gradle wrapper to 8.6
refactor(domain): extract money value object
```

### Configuration (`forge.local.md` `git:` section)

```yaml
git:
  commit_format: conventional             # conventional | project | none; default conventional
                                          # conventional: enforce Conventional Commits
                                          # project: defer to existing project hooks (set by hook detection)
                                          # none: no commit format enforcement
  commit_types:                           # full allowed list
    - feat
    - fix
    - chore
    - refactor
    - docs
    - test
    - perf
    - ci
  commit_scopes:                          # optional: restricts allowed scopes
    - domain
    - persistence
    - api
    - frontend
    - infra
  max_subject_length: 72                  # characters, default 72
  require_scope: false                    # default false
  sign_commits: false                     # gpg sign, default false
```

### Enforcement Rules

1. `{type}` must be from the configured `commit_types` list.
2. Subject uses imperative mood ("add", "fix", "remove" — not "added", "fixes", "removed").
3. Subject is lowercase (first character lowercase after `type(scope): `).
4. Subject has no trailing period.
5. Subject line length ≤ `max_subject_length`.
6. When `require_scope: true`, the `(scope)` segment is mandatory.
7. When `sign_commits: true`, commits are signed with `-S` flag.
8. When `commit_enforcement: external` (set by hook detection), forge defers entirely to the project's own tooling — no forge-level enforcement.

### NEVER

The following are strictly forbidden in any commit created by forge:

- **No `Co-Authored-By` trailers** — forge never adds AI attribution lines
- **No `Generated by` or similar AI attribution** in commit messages or bodies
- **No `--no-verify`** — hooks must not be bypassed
- **No `--force`** on commits already pushed to a shared branch

### Small Commit Strategy

Group changes by architectural layer to keep commits reviewable:

| Layer       | Typical contents                                  |
|-------------|---------------------------------------------------|
| domain      | Value objects, entities, domain events, errors    |
| use-case    | Application service, command/query handlers       |
| persistence | Repository impl, entity mapping, migrations       |
| api         | Controllers, DTOs, serializers, OpenAPI specs     |
| frontend    | UI components, stores, routes, styles             |
| infra       | Config, DI wiring, environment, build scripts     |

The implementer (`fg-300-implementer`) targets one commit per layer when changes span multiple layers. Single-layer changes collapse into one commit.

## Hook Detection

### Scanned Paths

During `/forge`, the following paths are scanned in the consuming project root:

| Path / Pattern              | Tool          |
|-----------------------------|---------------|
| `.husky/`                   | Husky         |
| `.git/hooks/commit-msg`     | Native git hook |
| `.pre-commit-config.yaml`   | pre-commit    |
| `lefthook.yml`              | Lefthook      |
| `commitlint.config.*`       | commitlint    |
| `.czrc` / `.cz.json`        | Commitizen    |

### Detection Algorithm

1. **Scan** — check each path in the table above for existence.
2. **Parse commitlint** — if `commitlint.config.*` is found, extract `types` and `scope-enum` values and write them into `forge.local.md` as `commit_types` and `commit_scopes` so forge uses the same constraints.
3. **Write to `forge.local.md`** — record detected tool names under `git.detected_hooks`.
4. **Set enforcement mode** — if any hook tool is detected, set `commit_enforcement: external`; otherwise set `commit_enforcement: forge`.

### Respect Rule

**Never override existing project hooks.** When `commit_enforcement: external` is set:
- Forge does not install its own `commit-msg` or `pre-commit` hooks.
- Forge does not validate commit messages itself — the project's tooling is authoritative.
- Forge passes `--no-edit` to merge commits only; never passes `--no-verify` to any commit.

## Consumers

| Consumer             | Uses                                                                           |
|----------------------|--------------------------------------------------------------------------------|
| `fg-100-orchestrator` | Reads `branch_template`, `ticket_source` to construct worktree branch names  |
| `fg-600-pr-builder`  | Reads `commit_format`, `commit_types`, `sign_commits`, `commit_enforcement`   |
| `/forge`        | Runs hook detection, writes `git:` section to `forge.local.md`               |
