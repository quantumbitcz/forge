# Data Classification

Defines secret detection, PII identification, and redaction rules for pipeline artifacts. Ensures no sensitive data leaks into `state.json`, stage notes, decision logs, or PR descriptions.

## Classification Tiers

| Tier | Label | Restriction | Pipeline Behavior |
|------|-------|-------------|-------------------|
| T1 | PUBLIC | No restriction | Stored and logged as-is |
| T2 | INTERNAL | No restriction, gitignored | Written only to `.forge/` (already gitignored) |
| T3 | CONFIDENTIAL | Redact from state.json, mask in logs | Replaced with `[REDACTED:SEC-SECRET-{NNN}]` in all pipeline artifacts |
| T4 | RESTRICTED | Block from artifacts, WARNING finding | Blocked entirely; emits SEC-SECRET CRITICAL finding and halts artifact write |

Tier assignment is automatic based on detection patterns. Overrides are not supported — a detected secret is always CONFIDENTIAL or RESTRICTED.

## Detection Patterns

L1 check engine regex patterns, evaluated on every `Edit`/`Write` via the PostToolUse hook.

### SEC-SECRET (CRITICAL)

API keys, tokens, passwords, and credential strings:

```regex
(api[_-]?key|apikey|secret[_-]?key|password|passwd|token|bearer)\s*[:=]\s*['"][^'"]{8,}
```

Private key blocks:

```regex
-----BEGIN (RSA |EC |DSA )?PRIVATE KEY-----
```

### SEC-PII (INFO)

Email addresses (personally identifiable information):

```regex
\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z]{2,}\b
```

PII detection is informational — it does not trigger redaction unless `pii_detection` is set to `redact` in configuration.

## Redaction Rules

When a CONFIDENTIAL or RESTRICTED value is detected in a pipeline artifact, the value is replaced before the artifact is written to disk.

### Redaction format

```
[REDACTED:SEC-SECRET-001]
```

The numeric suffix is a monotonically increasing counter scoped to the current pipeline run, stored in `state.json` at `data_classification.redaction_counter`.

### Affected artifacts

| Artifact | Path | Redacted |
|----------|------|----------|
| Pipeline state | `.forge/state.json` | Yes |
| Stage notes | `state.json → stage_notes.*` | Yes |
| Decision log | `.forge/decisions.jsonl` | Yes |
| Recap reports | `.forge/recap-*.md` | Yes |
| PR description | Generated by `fg-600-pr-builder` | Yes |
| Source code | Project files written by implementer | No (finding emitted, not redacted) |

Source code is excluded from automatic redaction because the implementer may be writing legitimate credential placeholders or test fixtures. Instead, a SEC-SECRET CRITICAL finding is emitted for human review.

### Redaction notification

When automatic redaction occurs, a SEC-REDACT INFO finding is emitted:

```
.forge/state.json:42 | SEC-REDACT-001 | INFO | Auto-redacted CONFIDENTIAL value | Review source of secret
```

## Check Engine Integration

The detection patterns above are added to the L1 rule set in `shared/checks/engine.sh`. They run as regex checks (sub-second) on every file write.

| Pattern | Category | Severity | Tier |
|---------|----------|----------|------|
| API keys/tokens/passwords | SEC-SECRET | CRITICAL | T3/T4 |
| Private key blocks | SEC-SECRET | CRITICAL | T4 |
| Email PII | SEC-PII | INFO | T1 |

Suppression via `rules-override.json` is supported with `"disabled": true` per rule ID, but a WARNING is logged when a SEC-SECRET rule is disabled.

## Finding Categories

| Category | Severity | Description |
|----------|----------|-------------|
| SEC-SECRET | CRITICAL | Detected credential, token, or private key |
| SEC-PII | INFO | Detected personally identifiable information |
| SEC-REDACT | INFO | Auto-redacted a CONFIDENTIAL/RESTRICTED value from an artifact |

All three categories fall under the existing `SEC-*` wildcard prefix in `scoring.md`. Scoring formula: `max(0, 100 - 20*CRITICAL - 5*WARNING - 2*INFO)`.

## Configuration

In `forge-config.md`:

    data_classification:
      enabled: true
      redact_artifacts: true
      custom_patterns: []
      pii_detection: true
      block_restricted: true

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `enabled` | boolean | `true` | Enable/disable data classification scanning |
| `redact_artifacts` | boolean | `true` | Auto-redact CONFIDENTIAL values in pipeline artifacts |
| `custom_patterns` | list | `[]` | Additional regex patterns (each entry: `{pattern, category, severity, tier}`) |
| `pii_detection` | boolean | `true` | Enable SEC-PII detection for email addresses |
| `block_restricted` | boolean | `true` | Block RESTRICTED-tier values from all artifacts |

## /forge-recover reset Behavior

`/forge-recover reset` clears `data_classification.redaction_counter` in `state.json`. Redaction history is not preserved across runs. `/forge-recover reset --hard` removes all artifacts including those containing `[REDACTED:...]` placeholders.

## 12. Input Classification (cross-reference)

This document governs *outbound* classification — what the pipeline writes (logs, PRs, artifacts) and how secrets are redacted on write. For *inbound* classification — how agents consume external data from MCP tools, wikis, caches, and cross-project learnings — see `shared/untrusted-envelope.md`. The two documents are complementary:

- Outbound (here): prevent secrets from leaving the pipeline.
- Inbound (envelope): prevent adversarial prompts from entering the pipeline.

All external data sources are tiered (Silent / Logged / Confirmed / Blocked) and wrapped in `<untrusted>` envelopes by `hooks/_py/mcp_response_filter.py` before reaching any agent. Credential-shaped content is quarantined at the filter layer and never reaches the envelope stage.

### Finding Categories addendum

| Category | Description |
|----------|-------------|
| `SEC-INJECTION-*` | Prompt-injection findings emitted by the inbound filter. See `shared/untrusted-envelope.md` and `shared/checks/category-registry.json` for the authoritative list and semantics. |
