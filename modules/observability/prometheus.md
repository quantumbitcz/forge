# Prometheus

## Overview

Prometheus is a pull-based time-series database and alerting system. It scrapes metric endpoints at configured intervals, stores data in its local TSDB, evaluates alerting rules, and integrates with Alertmanager for notification routing.

## Architecture Patterns

### Deployment topology
```
Services (expose /metrics)
  ← Prometheus (scrapes every 15–60s)
  → Alertmanager (firing rules)
  → Grafana (dashboards)
  → Remote write endpoint (Thanos, Cortex, Mimir for long-term storage)
```

Use **federation** only for aggregating pre-computed recording rules from multiple Prometheus instances — never federate raw high-cardinality metrics across clusters.

### Metric naming convention
```
{namespace}_{subsystem}_{name}_{unit}
```
Examples:
```
http_server_requests_total          # Counter — always _total suffix
http_server_request_duration_seconds # Histogram / Summary
process_resident_memory_bytes        # Gauge
db_connection_pool_active            # Gauge
job_failures_total                   # Counter
```

Rules:
- **Counters** always end in `_total`.
- **Histograms** end in `_bucket`, `_count`, `_sum` (auto-generated by client libraries).
- Use base units (`seconds`, `bytes`) not derived (`milliseconds`, `kilobytes`).
- Namespace = org or service group; subsystem = component. Both are optional but keep them consistent.

## Label Cardinality

Labels are the primary slice-and-dice mechanism — but every unique label-value combination creates a new time series.

**Hard limits:**
- Max unique label values per label: ~100 for low-traffic metrics; ~1,000 hard limit before cardinality explosion.
- Never use user IDs, request IDs, IP addresses, or free-form strings as label values.

**Good labels:** `method`, `status_code`, `route`, `region`, `environment`, `service`
**Bad labels:** `user_id`, `order_id`, `session_token`, `request_body`

Set `--storage.tsdb.retention.time` and monitor `prometheus_tsdb_head_series` — alert if it exceeds 1M active series.

## Histogram Bucket Selection

Buckets must match the latency distribution of the operation. Default Prometheus buckets (`[.005, .01, .025, .05, .1, .25, .5, 1, 2.5, 5, 10]`) work for HTTP; customize for other operations:

```yaml
# Slow background jobs (seconds to minutes)
buckets: [1, 5, 10, 30, 60, 120, 300]

# Fast cache operations (microseconds)
buckets: [0.0001, 0.0005, 0.001, 0.005, 0.01, 0.05, 0.1]
```

Place SLO thresholds at exact bucket boundaries so `histogram_quantile` produces accurate values at the SLO point.

## Recording Rules

Pre-compute expensive queries as new time series via recording rules — reduces query time for dashboards:

```yaml
groups:
  - name: http_slo
    interval: 60s
    rules:
      - record: job:http_server_requests_duration:p99_5m
        expr: histogram_quantile(0.99, sum(rate(http_server_request_duration_seconds_bucket[5m])) by (le, job))
      - record: job:http_server_requests:rate5m
        expr: sum(rate(http_server_requests_total[5m])) by (job, status_code)
```

Naming convention for recording rules: `{level}:{metric}:{operation}`.

## Alerting Rules

```yaml
groups:
  - name: availability
    rules:
      - alert: HighErrorRate
        expr: |
          (sum(rate(http_server_requests_total{status_code=~"5.."}[5m])) by (job)
          / sum(rate(http_server_requests_total[5m])) by (job)) > 0.01
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "{{ $labels.job }} error rate > 1%"
          runbook_url: "https://runbooks.example.com/high-error-rate"
```

Every alert must have: `severity` label, `summary` annotation, `runbook_url` annotation. Use `for:` to prevent flapping on transient spikes.

## Exemplars

Exemplars link a metric observation to a specific trace ID, enabling jump from a slow histogram bucket to the actual trace:

```
http_server_request_duration_seconds_bucket{le="0.5"} 42 # {trace_id="abc123"} 0.43
```

Requires Prometheus 2.26+ with `--enable-feature=exemplar-storage` and OpenMetrics format scraping. Client libraries must explicitly set exemplars on observations.

## Remote Write

Configure remote write for long-term storage (Thanos, Cortex, Mimir):

```yaml
remote_write:
  - url: https://thanos-receive.example.com/api/v1/receive
    bearer_token_file: /var/run/secrets/token
    queue_config:
      max_samples_per_send: 10000
      capacity: 50000
    write_relabel_configs:
      - source_labels: [__name__]
        regex: "go_.*"
        action: drop   # Drop Go runtime metrics to reduce volume
```

## Performance

- Scrape interval: 15s default; 60s for slow-moving metrics. Never below 5s.
- Target count per Prometheus instance: <10,000 targets for reliability.
- Series count: alert if `prometheus_tsdb_head_series > 1_000_000`.
- Use `scrape_timeout` shorter than `scrape_interval` to prevent cascade delays.
- Retention: 15 days local TSDB; offload older data to remote storage.

## Security

- Expose `/metrics` only on internal network interfaces or loopback; do not expose on the public internet.
- Use Prometheus `BasicAuth` or `TLSConfig` for scrape endpoints containing sensitive operational data.
- Apply network policies in Kubernetes so only the Prometheus pod can reach scrape targets on the metrics port.
- Sanitize label values — maliciously crafted label values can cause cardinality explosions (label injection attacks).

## Testing

- Validate recording and alerting rules with `promtool check rules rules.yml` in CI.
- Test alert expressions with `promtool test rules tests.yml` and crafted sample data.
- Verify metric names and label sets with integration tests that hit the `/metrics` endpoint and parse the output.
- Assert required metrics are present in smoke tests after deployment.

## Dos

- Follow the naming convention `{namespace}_{subsystem}_{name}_{unit}` consistently.
- Add `HELP` and `TYPE` comments to every custom metric via client library `Desc`.
- Use recording rules to pre-aggregate high-cost queries used in dashboards.
- Include a `runbook_url` in every alert annotation.
- Use exemplars to connect metric anomalies to traces.

## Don'ts

- Don't use high-cardinality values as label values.
- Don't scrape more frequently than 10s — Prometheus is not a real-time system.
- Don't rely on federation for raw metrics across clusters; use remote write instead.
- Don't create alerts without a `for:` duration — transient spikes will page on-call unnecessarily.
- Don't expose the Prometheus UI or `/metrics` on the public internet without authentication.
- Don't omit `_total` suffix on counters — it breaks convention and breaks dashboards expecting the suffix.
