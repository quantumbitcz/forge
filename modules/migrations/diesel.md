# Diesel Migrations Best Practices

## Overview
Diesel is a Rust ORM and query builder with a built-in migration system. Use it for Rust projects that need compile-time checked database interactions and a strong type system. Migrations are plain SQL files (`up.sql`/`down.sql`), keeping them portable and reviewable without Rust knowledge.

## Architecture Patterns

### Directory structure
```
migrations/
├── 00000000000000_diesel_initial_setup/
│   ├── up.sql
│   └── down.sql
├── 2024-01-01-120000_create_users/
│   ├── up.sql
│   └── down.sql
└── 2024-01-15-093000_add_email_index/
    ├── up.sql
    └── down.sql
src/
└── schema.rs       # Auto-generated by diesel print-schema
```

### Generate a new migration
```bash
diesel migration generate create_users
# Creates migrations/2024-01-01-120000_create_users/up.sql and down.sql
```

### up.sql / down.sql
```sql
-- migrations/2024-01-01-120000_create_users/up.sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- migrations/2024-01-01-120000_create_users/down.sql
DROP TABLE users;
```

### Apply and regenerate schema
```bash
diesel migration run          # applies pending migrations
diesel migration revert       # reverts last migration
diesel migration redo         # revert + run (tests down.sql + up.sql)
diesel print-schema > src/schema.rs  # regenerate after every migration
```

### Embed migrations in binary (recommended for production)
```rust
// In your main.rs or a db module
use diesel_migrations::{embed_migrations, EmbeddedMigrations, MigrationHarness};

pub const MIGRATIONS: EmbeddedMigrations = embed_migrations!("migrations");

pub fn run_migrations(conn: &mut PgConnection) {
    conn.run_pending_migrations(MIGRATIONS)
        .expect("Failed to run database migrations");
}
```
Call `run_migrations` at application startup before accepting requests.

### diesel.toml
```toml
[print_schema]
file = "src/schema.rs"
custom_type_derives = ["diesel::query_builder::QueryId", "Clone"]
```

## Configuration

### Cargo.toml
```toml
[dependencies]
diesel = { version = "2", features = ["postgres", "chrono", "uuid"] }
diesel-migrations = "2"

[dev-dependencies]
diesel = { version = "2", features = ["postgres"] }
```

### DATABASE_URL
```bash
# .env (development only — never commit real credentials)
DATABASE_URL=postgres://user:pass@localhost/myapp

# CI / production — inject via environment
export DATABASE_URL=postgres://user:pass@host/myapp
```

### CI setup
```bash
# Install diesel_cli (cache this in CI)
cargo install diesel_cli --no-default-features --features postgres

diesel setup
diesel migration run
cargo test
```

## Performance

### Zero-downtime: nullable column + backfill + constraint
```sql
-- up.sql (Deploy N): add nullable column
ALTER TABLE users ADD COLUMN display_name VARCHAR(255);

-- separate migration up.sql (Deploy N+1): backfill + enforce
UPDATE users SET display_name = name WHERE display_name IS NULL;
ALTER TABLE users ALTER COLUMN display_name SET NOT NULL;
```

### Large table index (Postgres concurrent)
```sql
-- up.sql
CREATE INDEX CONCURRENTLY idx_events_created_at ON events(created_at);

-- down.sql
DROP INDEX CONCURRENTLY idx_events_created_at;
```

### Data migrations with raw SQL
Diesel migrations are plain SQL — use `INSERT INTO ... SELECT` or `UPDATE` directly in `up.sql`. For complex logic in Rust, use `diesel::sql_query` in a migration run via embedded migrations with a custom runner.

## Security
- Store `DATABASE_URL` in environment variables; never commit credentials
- Use a dedicated migration user with DDL rights; app runtime gets DML only
- Commit all migration directories including `up.sql` and `down.sql`; treat applied migrations as immutable

## Testing
```bash
# Test each migration's round-trip in CI
diesel migration redo   # revert + re-apply every migration in sequence

# In Rust integration tests
#[test]
fn test_migrations_apply_cleanly() {
    let mut conn = PgConnection::establish(&database_url()).unwrap();
    conn.run_pending_migrations(MIGRATIONS).unwrap();
}
```
Use `diesel migration redo` in CI to verify both `up.sql` and `down.sql` are correct for every migration.

## Dos
- Always write `down.sql` — test it with `diesel migration redo` before merging
- Regenerate `schema.rs` after every migration and commit it to version control
- Embed migrations in the binary for production — removes the need for a separate migration step
- Use `diesel migration redo` in CI to verify round-trip correctness
- Keep `up.sql` and `down.sql` pure SQL — no Diesel Rust code in the migration files
- Run `diesel migration run` in a pre-startup hook when embedding is not possible

## Don'ts
- Never modify a migration directory after it has been applied to any environment
- Don't rely on `diesel migration revert` as a production rollback — have a DB snapshot strategy
- Avoid hand-editing `schema.rs` — it is always regenerated by `diesel print-schema`
- Don't mix DDL and large data operations in the same migration without considering lock contention
- Never commit `.env` files with real credentials to version control
- Avoid skipping `down.sql` — even a `SELECT 1;` placeholder is better than an empty file that silently succeeds rollback
